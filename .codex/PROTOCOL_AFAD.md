## §1 ARCHITECTURE

This protocol optimizes documentation for retrieval-augmented generation (RAG).

CORE INSIGHT: Documentation is not read linearly. It is chunked, embedded,
and retrieved. Every design decision answers: "Does this improve retrieval?"

Three retrieval properties govern all decisions:

    PROPERTY       MEANING                          OPTIMIZATION
    ──────────────────────────────────────────────────────────────────
    CHUNKABILITY   Split at semantic boundaries     One concept per entry
                                                    Hard cap at 600 tokens
    EMBEDDABILITY  Meaning captured in first        Keywords at entry start
                   100 tokens                       Avoid pronouns/forward refs
    ROUTABILITY    Query → correct file/entry       Distinctive terminology
                   without full-text search         Frontmatter hints (§4)

NOTE ON TOKEN LIMITS: The 600-token cap is not about LLM context capacity —
it is about embedding quality. A 600-token chunk embeds with more semantic
precision than a 2000-token chunk. The embedding vector for a bloated entry
diffuses toward the center of meaning and away from specifics.

RETRIEVAL COST MODEL:

    Cost = tokens_retrieved × retrieval_calls

    Goal: Minimize tokens while maximizing answer accuracy.
    Sweet spot: 1-3 retrieved entries answer 80% of queries.

ATOM RULES (applied by all schemas §6-§12):

- One concept per entry
- Self-contained (no "see above" or "see below")
- 200-400 tokens ideal; 600 max (INV-4)
- First sentence: what it IS (not what it does)
- Last section: deprecation if applicable

FILE RULES:

- One domain per file
- Critical entries at file start (first 30%)
- Edge cases at file end (last 20%)

Rationale: LLMs process start and end of context more reliably than middle
("lost in the middle" phenomenon). Position critical entries at edges.

LANGUAGE ADAPTATION:

This protocol is language-agnostic. Schema templates in §6-§12 use Python
syntax as an illustrative baseline. Adapt to the project language:

    PYTHON:      def method(arg: Type) -> Return:
    JAVA:        public ReturnType method(Type arg) throws X
    TYPESCRIPT:  method(arg: Type): Return
    GO:          func Method(arg Type) (Return, error)
    RUST:        fn method(arg: Type) -> Result<Return, Err>

Rules:
- Code block language tags MUST match the project language (```java, ```go, etc.)
- Schema structure (headings, sections, constraints) applies in all languages
- @property in Python maps to: getter methods in Java, computed properties in
  TypeScript/Kotlin, or method expressions in Go
- Type aliases (§9) map to: `type` in Python/TypeScript, interfaces or sealed
  types in Java, type aliases in Go/Rust. Omit §9 if the language has no equivalent.
- Test infrastructure (§12) terminology varies by framework — see §12 for mapping.

## §2 INVARIANTS

Five properties that must always hold:

    INV-1 COMPLETENESS   Every public export has exactly one doc entry
    INV-2 ACCURACY       signature_doc == signature_code
    INV-3 FRESHNESS      frontmatter.version == project.version
    INV-4 ATOMICITY      Every entry ≤600 tokens
    INV-5 CURRENT_STATE  Reference docs describe CURRENT API state only.
                         Historical provenance ("Added vX.Y", "(vX.Y+)")
                         belongs exclusively in CHANGELOG.md.
                         Deprecation notices are forward-looking only.

NOTE ON impl_hash: Previous versions included a sixth invariant requiring
an impl_hash comment per entry. This is now a RECOMMENDED PRACTICE, not
an invariant. The hash is generated by a CI tool or pre-commit hook — not
computed manually by an agent. Use `<!-- impl_hash: sha256:abc123 -->` when
available to accelerate sync loop (§17) semantic drift detection. Use
`<!-- impl_hash: manual -->` for behavior that cannot be hashed (side-effectful
IO, platform-dependent code). When absent, fall back to signature diff only.

Detection signals:

    SIGNAL          CONDITION                     ACTION
    ──────────────────────────────────────────────────────────────────
    ORPHAN-CODE     Export without entry          Create atom
    ORPHAN-DOC      Entry without export          Delete atom
    DRIFT           Signatures differ             Update + flag
    SEMANTIC-DRIFT  impl_hash differs             Flag Constraints for review
    BLOAT           Entry >600 tokens             Split into atoms
    VERSION-SCATTER "Added vX.Y" in docs/src      Remove; CHANGELOG.md only

## §3 FILE ARCHITECTURE

Two tiers, derived from filename pattern:

    PATTERN        TIER        CONTENT MODEL
    ──────────────────────────────────────────────────────────────────
    DOC_*.md       reference   Schema-driven, no prose
    *.md           auxiliary   Narrative permitted

Standard domains:

    FILE                DOMAIN      COVERAGE
    ──────────────────────────────────────────────────────────────────
    DOC_00_Index.md     INDEX       Exports, routing, navigation
    DOC_01_Core.md      CORE        High-frequency API (80% of queries)
    DOC_02_Types.md     TYPES       Data structures, type aliases
    DOC_03_*.md         PRIMARY     Main project domain
    DOC_04_*.md         SECONDARY   Supporting domain
    DOC_05_Errors.md    ERRORS      Exception hierarchy
    DOC_06_Testing.md   TESTING     Test infrastructure

DOC_00 INDEX FORMAT:

The index file MUST contain a routing table mapping symbol → file. Agents
query DOC_00 first to identify which file contains a given entry.

Minimum structure:

    | Symbol | File | Section |
    |:-------|:-----|:--------|
    | `ServiceClass` | DOC_01_Core.md | Core API |

Restructuring heuristics:

    ACTION  TRIGGER              JUSTIFICATION
    ──────────────────────────────────────────────────────────────────
    CREATE  >20 related exports  File exceeds retrieval sweet spot
    MERGE   <8 entries           File too small for dedicated embedding
    SPLIT   >60 entries          File exceeds context window efficiency

    Calibration: adjust thresholds ±30% based on token density.
    Dense entries (400+ tokens) → lower thresholds.
    Sparse entries (<200 tokens) → higher thresholds.

Scale guidance:

    SIZE           STRUCTURE                 NOTES
    ──────────────────────────────────────────────────────────────────
    Small (<20)    DOC_00 + DOC_01 only      Domains collapsed
    Medium (20-80) Standard (DOC_00-06)      One domain per file
    Large (>80)    + domain splits           Sub-indices considered

## §4 FRONTMATTER

Required fields for every doc:

```yaml
---
afad: "3.5"
version: "X.Y.Z"
domain: CORE
updated: "YYYY-MM-DD"
route:
  keywords: [terms, that, find, this]       # see guidance below
  questions: ["natural language query"]     # see guidance below
---
```

Field semantics:

    FIELD    PURPOSE                    NOTES
    ──────────────────────────────────────────────────────────────────
    afad     Schema version             Must match current protocol version
    version  Project version (SemVer)   Freshness check (INV-3)
    domain   Semantic cluster           UPPERCASE; match standard domains in §3
    updated  Last modification date     Format: "YYYY-MM-DD" (ISO-8601)
    route    Retrieval hints            keywords + questions (see below)

ROUTE GUIDANCE:

Keywords and questions exist for the disambiguation case: two files with
similar content and ambiguous embedding distance. They are NOT a substitute
for thin or generic content — if the content is weak, fix the content.

- `keywords`: 5-10 distinctive terms not likely to appear in other files.
  Avoid generic words ("function", "class", "method", "handler").
  If you cannot find 5 distinctive terms, the file may be too broad.

- `questions`: 2-5 natural language queries this file uniquely answers.
  Bad: "how do I use functions?"
  Good: "how does Registry.resolve handle missing keys?"
  Questions MUST be unique across files; duplicates cause misdirection.

Both fields MAY be omitted for files with highly distinctive content that
will not collide with other files during retrieval. When in doubt, include.

## §5 SCHEMA SELECTION

    Component type?
    │
    ├─ Callable (function/method/class) ──→ §6
    ├─ Property (@property) ──────────────→ §7
    ├─ Enum ──────────────────────────────→ §8
    ├─ Type alias ────────────────────────→ §9
    ├─ Constant ──────────────────────────→ §10
    ├─ Exception ─────────────────────────→ §11
    └─ Fixture / Marker / Hook ───────────→ §12

All schemas share:

- `## \`name\`` heading (backticks required)
- `### Signature` code block with language tag
- `### Constraints` semantic list
- `---` separator between entries

WHY SEPARATE SCHEMAS: Properties take no parameters. Enums have a Members
table. Exceptions have a Recovery field. Constants have no parameters or
return value. Collapsing these into one schema pushes structural decisions
to the agent on every use — exactly the kind of ambiguity a protocol exists
to eliminate.

## §6 CALLABLE SCHEMA

For functions, methods, classmethods, staticmethods.

Template (~~~ outer fence allows inner ```language blocks):

~~~markdown
## `function_name`

Function/method that <does X>.   ← EMBEDDABILITY: first line = what it IS

### Signature
```python
def function_name(required: Type, /, optional: Type = default) -> Return:
```

### Parameters
| Name | Req | Semantics |
|:-----|:----|:----------|
| `required` | Y | Fragment ≤10 words |
| `optional` | N | Fragment ≤10 words |

### Constraints
- Return: What is produced; edge cases (None, empty, sentinel values)
- Raises: Specific conditions; "Never raises" when applicable
- State: Mutates X | Read-only | Pure
- Thread: Safe | Unsafe
- Complexity: O(time) / O(space)   ← OPTIONAL: algorithmic cores only

### Usage   ← OPTIONAL: decision guidance when similar APIs exist
- When: Condition where this is preferred
- Prefer: Alternative and when
- Avoid: Anti-pattern with reason

### Example   ← OPTIONAL: minimal one-shot usage (≤5 lines)
```python
result = function_name(arg1, arg2)
```

### Notes   ← OPTIONAL: quirks, deprecation
- Deprecated: vX.Y. Use Y instead. Removal: vZ.0.

---
~~~

PARAMETER TABLE RULES:
- `Name`: exact parameter name, in backticks
- `Req`: "Y" or "N" only — do not expand, do not omit
- `Semantics`: fragment (no articles), ≤10 words
- Type is NOT a column — it is already in the Signature. Repeating it
  wastes tokens and creates a second source of drift.

CONSTRAINTS FORMAT:
Use plain bullets. The pipe-delimited format (`Raises: X when Y | Never`)
is idiomatic for Python/Java but breaks down for other error models:
- Rust:  "Returns: `Err(NotFound)` when key not registered"
- Go:    "Returns: `(nil, ErrNotFound)` when key not registered"
- Swift: "Throws: `NotFoundError` when key not registered"
Write naturally for the project language.

NOTES RULES:
- NO "Added vX.Y" — historical provenance belongs in CHANGELOG.md (INV-5)
- Deprecation notices ARE permitted (forward-looking, affects user decisions)
- Deprecation format: "Deprecated: vX.Y. Use <alternative>. Removal: vZ.0."

TOKEN TARGET: 200-400 tokens. If >500: split (class entry separate from method entries).

OVERLOADED FUNCTIONS: Include ALL @overload signatures. Parameters documents the union.

COMPLEXITY:
- REQUIRED for: parsing functions, IO operations, search algorithms
- OPTIONAL for: simple accessors, delegation methods, constructors
- Omit entirely for trivially O(1) operations

## §13 AUXILIARY DOCUMENTATION

Applies to any `*.md` that does NOT match `docs/DOC_*.md`.

### GUIDE STRUCTURE

~~~markdown
---
afad: "3.5"
version: "X.Y.Z"
domain: <TOPIC>
updated: "YYYY-MM-DD"
route:
  keywords: [topic keywords]
  questions: ["how do I X?"]
---

# <Topic> Guide

**Purpose**: What reader learns
**Prerequisites**: Required knowledge

## Overview

2-3 paragraphs of context.

## <Action Step>

Instruction.

```python
# Working, runnable code
```

## Summary

- Key point 1
- Key point 2
~~~

FORMATTING RULES:
- All code blocks MUST specify a language tag (```python, ```bash, ```yaml, etc.)
- All code blocks MUST be runnable (no pseudo-code, no placeholder ellipsis)
- No decorative emoji (see §15)
- Economy of words: no filler phrases ("It is important to note...", "As mentioned above...")
- No full API signatures — those belong in DOC_*.md reference files; link instead

### README

README.md is a storefront. Permitted: installation, quick start, examples, links.
Prohibited: full API documentation, exhaustive parameter lists.
All code snippets must be runnable as shown.

YAML frontmatter is PROHIBITED for README.md: source control platforms render
it as a visible table, breaking user-facing presentation. Use an HTML comment
block at the top if retrieval metadata is needed:

```html
<!--
RETRIEVAL_HINTS:
  keywords: [topic keywords]
  answers: [user questions answered here]
  related: [related doc paths]
-->
```

### QUICK REFERENCE

Task-oriented, copy-paste, zero prose. Every snippet runs independently.
Max 2 sentences per task category. No frontmatter required.

## Fuzzer Summary

| Fuzzer | Target Module(s) | Patterns | Seeds | Concern |
|:-------|:-----------------|:---------|:------|:--------|

## Module Coverage Matrix

| Source Module | Fuzzers Covering It |
|:--------------|:--------------------|
| `mod.sub` | fuzz_X, fuzz_Y |

## `fuzz_X`

Target: `module.path` (1-line what it tests)

| Pattern | Weight | Invariants Checked |

~~~

RULES:

- Module Coverage Matrix: derived from fuzzer imports; enables gap analysis

- No prose paragraphs; tables and fragments only
- Seeds column: count of seed files for the target
- Fuzzer naming convention adapts to language: `fuzz_X.py` (Python/atheris),
  `FuzzX.java` (Java/JQF), `fuzz_x.ts` (JS/Jazzer.js) — use the project convention

## §15 ANTI-PATTERNS

    ANTI-PATTERN              WHY WRONG                        FIX
    ──────────────────────────────────────────────────────────────────────
    Type in parameter table   Duplicates signature; two        Name + Req +

    Prose in Parameters       Wastes tokens                    Fragments only
    Duplicating docstrings    Double maintenance               Reference only
    Complex examples in       Bloats chunks                    examples/ dir or
    reference atoms                                            auxiliary guides
    "See above" references    Breaks atomicity                 Self-contained
    >600 token entries        Embedding quality degrades       Split atoms
    Generic route keywords    Poor disambiguation              Distinctive terms
    "Added vX.Y" anywhere     Wrong location (INV-5)           CHANGELOG.md only

                              toolchain rendering
    Full API in aux docs      Duplication                      DOC_*.md only

    PERMITTED: Minimal one-shot examples in reference atoms (≤5 lines,
    self-contained, no imports beyond stdlib).

## §16 LIFECYCLE

DEPRECATION (3-phase, minimum 2 minor versions):

    PHASE         LOCATION             CONTENT
    ──────────────────────────────────────────────────────────────────────
    ANNOUNCE      ### Notes            Deprecated: Use X. Removal: vN+2.
    (version N)

    WARNING       ### Constraints      Deprecated: vN. Migration: §link.
    (version N+1)

    REMOVAL       Delete entry         CHANGELOG: Removed section.
    (version N+2)

DOCUMENTATION VERSIONING:

    CHANGE TYPE                   AFAD VERSION   EXAMPLE
    ──────────────────────────────────────────────────────────────────────
    New optional field or column  Minor bump     afad: "3.5"
    New required field            Major bump     afad: "4.0" adds required X
    Schema restructure            Major bump     afad: "4.0" renames section
    Required field removal        Major bump     afad: "4.0" removes field

## §17 MAINTENANCE

SYNC LOOP (on code change):

    1. INVENTORY: Parse exports → build symbol:signature map
    2. COMPARE: For each symbol, categorize:
       - MATCH → no action
       - DRIFT → update Signature; flag Parameters for review
       - SEMANTIC-DRIFT → impl_hash differs; flag Constraints for review
       - ORPHAN-CODE → create atom
       - ORPHAN-DOC → check for MOVE (same name + matching signature),
                      else delete atom
    3. RECONCILE: Apply categorized actions
    4. LINK-CHECK: Validate backtick references in all *.md
       (`symbol` in *.md → symbol exists in exports)
    5. FINALIZE: Update frontmatter, verify invariants, commit

REFACTORING HEURISTICS:

    ORPHAN-DOC(A) + ORPHAN-CODE(B) with matching name + matching signature:
    → Classify as MOVE (not Delete + Create)
    → Transfer Notes, Usage, Examples, impl_hash history from A to B

AUTONOMOUS AUTHORITY (agents may act without permission):

    ACTION  TRIGGER              REQUIREMENT
    ──────────────────────────────────────────────────────────────────────
    DELETE  All atoms removed    Log in commit
    MERGE   <8 entries           Log in commit
    SPLIT   >60 entries          Log in commit
    CREATE  >20 new symbols      Log in commit

CO-EVOLUTION RULE:

    Docs and code change in the same commit. A code change without a doc
    update is an invariant violation — treat it the same as a drift error.

    Scope: public API changes, invariant changes, architectural boundary
    changes, and tooling changes that affect documented behavior (e.g.
    test runner, lint config).

## §18 VALIDATION

    LEVEL  CHECK                         BLOCKING
    ──────────────────────────────────────────────────────────────────────
    L0     Frontmatter valid YAML        Yes
    L0     Required fields present       Yes
    L1     Every entry has Signature     Yes
    L1     Every entry has Constraints   Yes
    L2     Signatures match code         Yes
    L2     No orphan symbols             Yes
    L2     Entries ≤600 tokens           Yes
    L2     Referential integrity         Yes
           (`symbol` in *.md → exists in exports)
    L3     Parameters uses Y/N           No
    L3     Fragments ≤10 words           No
    L3     Route keywords distinct       No (warn if in >3 files)
    L3     Route questions unique        No (warn if duplicate)

RECOVERY:

    FAILURE  RECOVERY
    ──────────────────────────────────────────────────────────────────────
    L0       Fix YAML, add missing fields
    L1       Add Signature/Constraints with minimal content
    L2       Run sync loop; split bloated entries
    L3       Fix in subsequent pass

## §19 CONFLICT RESOLUTION

Priority: P0 Accuracy > P1 Completeness > P2 Structure > P3 Style

    CONFLICT                            RESOLUTION
    ──────────────────────────────────────────────────────────────────────
    Complex signature doesn't fit       Keep accurate signature (P0)
    standard Parameters table           Add Notes explaining structure

    Entry exceeds 600 tokens            Split into atoms (P2)
    but splitting loses cohesion        unless accuracy suffers (P0 wins)

    Undocumented export,                Create placeholder (P1)
    semantics unknown                   with TODO in Constraints

    Style violation,                    Fix style (P3) in next pass
    content is accurate                 Don't block on style

## §20 WORKED EXAMPLES

Generic callable schema applied to a hypothetical `Registry.resolve`.
Two language variants shown — Python and Java — to illustrate adaptation.
Note the parameter table has no Type column; types live in the Signature only.

~~~markdown
## `Registry.resolve`

Method that resolves a registered item by key.

### Signature
```python
def resolve(
    self,
    key: str,
    /,
    *,
    strict: bool = True,
) -> Item | None:
```

### Parameters
| Name | Req | Semantics |
|:-----|:----|:----------|
| `key` | Y | Registration key; must be non-empty |
| `strict` | N | Raise on missing key when True |

### Constraints
- Return: Registered `Item`, or `None` when `strict=False` and key absent
- Raises: `KeyError` when `strict=True` and key not registered; never when `strict=False`
- State: Read-only
- Thread: Safe for concurrent reads

---
~~~

Token count: ~110 (within 200-400 target)
First line: States what method IS
Self-contained: No cross-references required
No Type column: `str`, `bool`, `Item | None` are already in the Signature

### Parameters

|:-----|:----|:----------|
| `key` | Y | Registration key; must be non-blank |
| `strict` | N | Throw on missing key when true |

### Constraints
- Return: Registered `Item`, or `null` when `strict=false` and key absent
